## Folder layout

> Note: this project keeps its workspace **inside** the MasterMind repo, under `workspace/` (a deliberate choice — differs from the Core Rule's default of keeping the workspace outside the repo).

```
MasterMind/
├── core/ · models/    ← skill library
└── workspace/         ← this BA project (committed into the repo)
    ├── input/             ← raw materials (SRS docx, CR files, screenshots…)
    ├── context/           ← .md sidecars auto-generated by the agent
    ├── output/            ← deliverables (.docx, .xlsx, .drawio, .json…)
    ├── docs/              ← artifact templates the user works on directly
    └── conventions/       ← project overrides for skill defaults
```

## Standard workflow

1. **Drop raw material** into `input/` (any file format)
2. **Agent reads + creates `context/<file>.md`** sidecar (Core Rule Layer 1)
3. **User opens `docs/<artifact>.md`** and asks the agent to fill the content
4. **Agent uses the appropriate model_001 skill** — never invents the format
5. **Final deliverable** is written to `output/` (e.g. `.docx`, `.xlsx`)

## Common commands

```
"sync drive"           → pull latest from Google Drive into context/ (if the project has Drive sync set up)
"create story under FEAT-XX"  → agent uses the google_sheets API or edits docs/backlog.md
"gen Jira tasks"       → agent uses skill integration/jira → output/jira/*.md
"build SRS"            → agent uses skill document/srs → output/<project>-SRS.docx
"render ERD .drawio"   → agent renders Mermaid → output/<project>-ERD.drawio
```

## Core Rule reminders

- ✅ `input/` = user-managed (drop files here, the agent must not write to it)
- ✅ `context/` = agent-managed (md sidecars, do not edit manually — overwritten on next sync)
- ✅ `output/` = deliverables — the agent writes, the user reads/exports
- ✅ Every binary file in `output/` must have a `.md` sidecar in `context/` (token-cheap reading)
- ❌ Do not edit `MasterMind/` — it is the skill library, synced from GitHub

# Requirements table — structure

The **single consolidated table** every business analysis produces. Whatever the input format (documents, notes, transcripts, spreadsheets, diagrams), all requirements land here.

## Columns

| Column | Meaning |
|---|---|
| **Req code** | Requirement identifier, format `REQ-xxxx` (4 digits, e.g. `REQ-0007`). Sequential, globally unique, never reused. **Auto-generated** — never hand-assigned. |
| **Topic** | The common group a requirement belongs to. Groups related requirements together (e.g. `Authentication`, `Billing`, `Reporting`). |
| **Criteria** | A sub-topic within the Topic — effectively the requirement's **title**: specific, short, scannable. |
| **Description** | The detailed description of the requirement. |
| **Ref. Docs** | The cited source. If the source has headings/sections, name them in detail (e.g. `Vendor SRS §4.2 "Login flow"`) so the reader can locate it. |
| **Q&A** | *Optional* — left for the user to add follow-up questions and answers later. |
| **Remarks** | *Optional* — left for the user to discuss / annotate on the document later. |

## Timestamp batching

Requirements are **saved in timestamp groups** for later traceability. The timestamp is the moment the user **requested the analysis** of a piece of input.

- Each analysis run = one `##` section, headed with its timestamp and a short source description.
- `REQ-xxxx` codes are **globally sequential** across all runs — a code is never reused, so any requirement traces back to exactly one run.
- A re-run on new input **appends a new section**; existing sections are never rewritten.

## File form — `context/.../requirements/context.md`

```markdown
# Requirements

> Each section is one analysis run, tagged with the timestamp it was requested.
> REQ codes are globally sequential and never reused.

## 2026-05-21 14:30 — Vendor SRS v2 + kickoff notes

| Req code | Topic | Criteria | Description | Ref. Docs | Q&A | Remarks |
|---|---|---|---|---|---|---|
| REQ-0001 | Authentication | Email + password login | The user signs in with a registered email and password. | Vendor SRS §3.1 "Sign in" | | |
| REQ-0002 | Authentication | Password reset | The user resets a forgotten password via an emailed link. | Vendor SRS §3.2 | | |
| REQ-0003 | Billing | Monthly invoice | The system issues one invoice per account per month. | Kickoff notes, "Billing" topic | | |

## 2026-05-28 09:15 — Stakeholder interview round 2

| Req code | Topic | Criteria | Description | Ref. Docs | Q&A | Remarks |
|---|---|---|---|---|---|---|
| REQ-0004 | Reporting | Export to CSV | The user exports any report table to CSV. | Interview transcript, Q14 | | |
```

## Output

- **Context (`.md`)** — the file above; the source of truth.
- **Pilot (`.xlsx`)** — generated by [`scripts/ba_md_to_xlsx.py`](../scripts/) into `output/`. The `.xlsx` flattens all run sections into one sheet and prepends a **`Run`** column (the section timestamp) so traceability survives into the pilot copy. `Q&A` and `Remarks` are left empty for the user.

## Rules

- One requirement = one row. Keep `Criteria` short; put detail in `Description`.
- Never leave `Ref. Docs` empty — every requirement must be traceable to a source.
- Never hand-write `Req code` — the generator assigns the next sequential code.
- Topic values should be reused, not reinvented per row — they are the grouping key.
